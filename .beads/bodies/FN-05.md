# FN-05 — Supply cache spawn (rejection sampling against patrol)

## Summary
Implement the supply-cache mechanic: at game start, spawn one cache at a random empty cell (rejection-sample against patrol segments). On patrol-head collision with the cache cell, increment score by 1, call `patrol.grow(1)`, and immediately spawn a new cache on the next tick (per PRD §3.34, no delay). Always exactly one cache on board at a time.

Per PRD §3.34 (no respawn delay); §4.1 (score is monotonic); §4.4 (growth +1 on cache); §13.6 (cache never spawns on patrol or bunkers — bunkers come in Phase 4, handled then via P4-05).

## Persona
Primary: End User (cache pickup is the core scoring loop).
Secondary: System (spawn rejection-sampling).

Call once at game start. After `patrolStep()`, if `head.x === supplyCache.x && head.y === supplyCache.y`, do `state.score += 1`, `patrol.grow(1)`, then `spawnSupplyCache()`. ## Acceptance Criteria
### End User
- [ ] One supply cache is visible on board at game start, not on top of the patrol.
- [ ] Steering the patrol head into the cache increments score by 1 and grows the patrol by 1 segment.
- [ ] A new cache appears immediately on the same tick the old one is consumed.

### System
- [ ] `spawnSupplyCache` never places the cache on a cell currently occupied by a patrol segment.
- [ ] Exactly one cache exists at any time.

## Patterns to Reuse
- Rejection sampling: `do { x = randInt(0, 32); y = randInt(0, 24); } while (patrolOccupies(x, y) && --attempts > 0);` Future bunker/wormhole exclusion will extend the occupancy check via P4-05.

## Testing Strategy

- Manual edge: run for ~30 s gathering caches, verify length grows monotonically.
