# P1-03 — Intra-phase speed ramp (6 → 8 cells/sec by score 50)

## Summary
Implement the in-Phase-1 speed ramp: `tickRate` scales with score from 6 cells/sec at score 0 to 8 cells/sec at score 50 (the Phase 1→2 gate). The PRD specifies "one cell/sec per 25 score" (§3.16) so the formula is `tickRate = 6 + floor(score / 25)`, capped at 8 within Phase 1. Phase transitions re-baseline; later phases will extend the formula (P5 sectors multiply on top per §3.11, handled by P5-03).

Per PRD §3.16, §5.3, §12.3.

## Persona
Primary: End User (perceptible difficulty curve).

Phase-1 formula: `state.tickRate = Math.min(6 + Math.floor(state.score / 25), 8)`. Hook on phase entry too (so re-baseline works when phases re-set their base via P2-01 etc.). |

## Acceptance Criteria
### End User
- [ ] At score 0 in Phase 1, patrol moves at ~6 cells/sec (perceptibly slow).
- [ ] At score 25, patrol moves at ~7 cells/sec (slightly faster).
- [ ] At score 50 (right at the Phase 1→2 gate), patrol moves at ~8 cells/sec.

### System
- [ ] `tickRate` is recomputed on every score increment.
- [ ] In Phase 1, `tickRate` is clamped at 8 — score above 50 (which shouldn't happen before the gate but is possible if the gate fires after score) doesn't push beyond 8 within Phase 1.
- [ ] Other phases will override / extend the formula via their own `recomputeTickRate` branch — this bead establishes the Phase-1 branch only.

## Patterns to Reuse
- Phase-branched formula inside `recomputeTickRate()` — like `palette()`, a single source of truth that switches on `state.phase`.

## Testing Strategy
- Manual: collect supply caches up to 25 score, time the difference (subjective: it's perceptibly faster).
- Console probe: set `state.score = 50; recomputeTickRate(); console.log(state.tickRate)` — expect 8.
- Console probe: set `state.score = 100; recomputeTickRate(); console.log(state.tickRate)` while still in Phase 1 — expect 8 (clamped).
- Regression: tick rate base 6 at score 0 still holds; FN-02 loop respects the value.
