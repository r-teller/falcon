# P1-02 — Wall and self-collision loss

## Summary
After each patrol step, check whether the head landed on (a) a cell outside the grid (wall) or (b) a cell occupied by any non-head patrol segment (self). Either case calls `losePatrol()` (provided by FN-08), which displays the PATROL LOST overlay. Walls kill across all phases per PRD §3.17.

For Phase 1: any loss is terminal (no reinforcements yet; FIN-02 adds them later). For now `losePatrol()` always shows the overlay.

Per PRD §3.17 (walls kill), §4.5 (wall + self lose patrol).

## Persona
Primary: End User (collision death is core feedback).

## Changes Needed
| File | Action | What Changes |

On non-null result, call `losePatrol()` (FN-08) and skip further per-tick processing this tick. |

## Acceptance Criteria
### End User
- [ ] Driving the patrol into the top, bottom, left, or right edge triggers the PATROL LOST screen.
- [ ] Driving the head into a body segment triggers the PATROL LOST screen.
- [ ] After a collision, the patrol stops moving and the overlay appears immediately (no further ticks).
- [ ] Cells INSIDE the grid (0 ≤ x ≤ 31, 0 ≤ y ≤ 23) do not trigger wall loss.

### System
- [ ] Collision detection runs AFTER `patrolStep()` (head has already advanced) — verifies the post-step head position.
- [ ] Self-collision check uses `segments.slice(1)` (skip head) to avoid false positives.

## Patterns to Reuse
- Simple bounds check + linear scan of segments. At max patrol length (~150 segments late game), linear scan is fine; switch to a Set if profiling demands.

## Testing Strategy
- Manual: drive into each wall, verify loss + overlay.
- Manual: grow patrol to ~5 segments, deliberately turn into body, verify self-collision loss.
- Regression: normal movement (no collision) keeps running.
