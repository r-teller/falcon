Methodology & Data Lineage

Schema reference: .claude/docs/metrics-schema.md. Validator: .claude/schemas/metrics.schema.json. Source: forward-looking emission from .claude/scripts/token-tracking.sh (per-bead per-phase records via start/stop at /leroy Step 3e + /wrapup Task 3c flush). Post-hoc reconstruction also supported via .claude/scripts/inject_model_breakdown.py (model usage from transcripts) and project-side reconstruct_metrics.py scripts.

Data sources

• Bead catalog — bd list --json (open + closed beads with full bodies including Effort Forecast).
• Forward-looking emission — token-tracking.sh bookmarks transcript line offsets at start --bead <id> --phase <phase> and computes deltas at stop. Per-session, per-bead, per-phase records.
• Coordinate-phase — token-tracking.sh --coordinate --beads <id1,id2,…> captures multi-bead planning cost separately, with allocation_method: equal_split.
• Post-hoc model breakdown — inject_model_breakdown.py reads session transcripts, extracts model field per assistant turn, emits one record_type: model_breakdown record per session per model. Required for the per-model cost dashboards.
• Sprint definition — the branch field on each record (derived from git rev-parse --abbrev-ref HEAD at session start).

Phase boundaries

Live emission: each start/stop boundary is explicit — the user/agent transitions phases via stop --bead <id> then start --bead <id> --phase <next>. Phase enum: plan / discover / implement / test / fix (5 work phases) plus coordinate for multi-bead planning.
Post-hoc reconstruction (project-side scripts only): tool-call pattern classification — Read/Grep = plan, Edit/Write = implement, Bash(pytest|playwright|tsc|…) = test, Edit-after-fail = fix. ~10–20% boundary noise. LLM-judge refinement optional.

Origin classification

Each (session, bead) row gets an origin from handoff.yaml: committed if in the sprint's completed/in_progress/next_steps; discovered if in discovered; carryover if it was in_progress in the prior sprint; added otherwise (with explanatory note).

Time bucketing

Session-time records (record_type: session_time) use project-adaptive percentile thresholds for gap classification (default P75 / P93 / P97.7 cuts). Computed from the project's own gap distribution — tighter than uniform defaults because agentic-dev sessions have rapid back-to-back exchanges. Override thresholds in your wrapup workflow if your project pattern differs.

Coverage caveats

• Sessions that never invoke bd update -s in_progress via /leroy Step 3e produce no per-bead records — ad-hoc / exploratory work is invisible to this dashboard.
• Beads updated outside the /leroy claim flow (direct bd update / bd close) won't have effort records. /wrapup Task 3c surfaces these as warnings (orphaned closures) so you can decide whether to backfill.
• forecast_* values come from parsing the Effort Forecast section of each bead's body (per the work-item-templates.md contract: 5-phase breakdown — plan/discover/implement/test/fix — plus Total + Confidence). Set on each phase's first start to avoid multi-session double-counting on resume.
• Sprint dates use the latest stopped timestamp per branch. Multiple branches can end on the same day — not duplicates.

Vocabulary

• bead = one work item / task in the bd tracker
• turn = one assistant message in the conversation (1 turn ≈ 1 round-trip)
• token = ~3–4 characters of model input/output
• sprint = one PR branch worth of work (typically 1–N sessions per sprint)
• phase = work-shape label on each record. Five work phases (plan / discover / implement / test / fix) plus coordinate for multi-bead planning.
• flow efficiency = % of session wall-clock during which the model was generating, vs waiting for human input
• Cynefin domain = complexity classification: clear (repeatable), complicated (expert-required), complex (experimental), chaotic (no causal model)
